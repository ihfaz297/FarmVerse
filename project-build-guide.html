Project A

Vision-Guided Object Sorting Arm

A fixed camera runs pretrained YOLOv8n over a tray, a laptop maps each detection to one of three physical zones, and a 4-DOF servo arm executes a pre-taught pick-and-place sequence into the matching bin. No inverse kinematics, no custom-trained model — the engineering is in the zone mapping, the serial handshake, and grasp reliability.

Architecture

Webcam → YOLOv8n inference (Python, laptop) → stable detection for 10+ consecutive frames → centroid mapped to nearest of 3 fixed zones → single-byte command over USB serial → Arduino/ESP32 runs the matching waypoint sequence (home → hover → lower → grip → lift → carry → release → home) → done-ack sent back → laptop waits for ack before issuing the next command.

8-day timeline

01

Order & assemble

Order every part today — shipping is your biggest untracked risk. Assemble the arm mechanically per kit instructions. No electronics yet: confirm every joint moves freely by hand first.

02

Wire & find waypoints

Wire servos through the PCA9685 with external power, common ground to the MCU. Sweep-test each servo for jitter/brownout. Hand-tune and record angle sets for: home, each zone's pick position, each bin's drop position, gripper-open, gripper-closed. Write these numbers down — this list is your most important artifact.

03

Pick-place sequence + grasp smoke test

Write the Arduino routine that runs a full pick-to-bin sequence on a single serial byte. Trigger it manually from the Serial Monitor. Gate: 10 consecutive successful grasps on one zone before moving on. If the gripper is unreliable, fix it now — add a rubber/foam pad, adjust the closed angle — don't defer it.

04

Get detection working

Stand up the YOLOv8n webcam script, draw live bounding boxes, print class + centroid to console. Verify it reliably detects your actual demo objects, under your actual demo lighting.

05

Zone mapping + serial link

Map centroid pixel coordinates to the 3 fixed zones. Require a stable detection across ~10 frames before sending a command, to kill false triggers from flicker. Wire up pyserial to send the byte and wait for the done-ack.

06

Full integration

Object in a zone → detected → correct byte sent → correct arm sequence runs → object lands in the correct bin. Debug timing so the laptop never sends a second command before the arm acks the first. Log 15–20 full cycles across all 3 zones; note the success rate.

07

Robustness pass

Handle no-detection frames as an idle state, not an error. Add a retry-or-skip path for a likely-failed grasp. Tidy the wiring into something that survives being carried to the demo room. Record a backup demo video in case live hardware hiccups.

08

Buffer + polish

Final test pass. Write the GitHub README (wiring photo, circuit diagram, demo GIF). Rehearse the viva: why fixed zones instead of inverse kinematics, how the serial handshake avoids race conditions, and the future-work framing toward learned grasp models.

Code skeletons

Serial protocol: laptop sends a single ASCII digit for the target zone; Arduino replies "D\n" once the full sequence completes. Nothing more elaborate is needed for 3 zones.

// arduino_arm.ino — zone command -> waypoint sequence
#include <Servo.h>
Servo base, shoulder, wrist, gripper;

struct Pose { int base, shoulder, wrist, gripper; };
Pose HOME       = {90, 90, 90, 60};
Pose ZONE_PICK[3] = { {40,120,70,60}, {90,120,70,60}, {140,120,70,60} };
Pose BIN_DROP[3]  = { {40,60,110,60}, {90,60,110,60}, {140,60,110,60} };
const int GRIP_OPEN = 60, GRIP_CLOSED = 140;

void moveTo(Pose p) {
  base.write(p.base); shoulder.write(p.shoulder); wrist.write(p.wrist);
  delay(400); // let servos settle before the next command
}

void pickAndPlace(int zone) {
  moveTo(ZONE_PICK[zone]);
  gripper.write(GRIP_CLOSED); delay(300);
  moveTo(HOME);
  moveTo(BIN_DROP[zone]);
  gripper.write(GRIP_OPEN); delay(300);
  moveTo(HOME);
}

void setup() {
  Serial.begin(9600);
  base.attach(13); shoulder.attach(12); wrist.attach(11); gripper.attach(10);
  moveTo(HOME);
}

void loop() {
  if (Serial.available()) {
    int zone = Serial.parseInt();
    if (zone >= 0 && zone <= 2) {
      pickAndPlace(zone);
      Serial.println("D"); // done-ack: laptop waits for this
    }
  }
}
# detect_and_send.py — YOLO detection -> zone -> serial byte
from ultralytics import YOLO
import cv2, serial, time

model = YOLO("yolov8n.pt")
ser = serial.Serial("COM5", 9600, timeout=2)  # adjust port
cap = cv2.VideoCapture(0)
ZONE_BOUNDS = [(0, 213), (213, 426), (426, 640)]  # 3 vertical bands, 640px frame
stable_zone, stable_count = None, 0

def zone_for_x(cx):
    for i, (lo, hi) in enumerate(ZONE_BOUNDS):
        if lo <= cx < hi:
            return i
    return None

while True:
    ok, frame = cap.read()
    if not ok: break
    results = model(frame, verbose=False)[0]
    target_zone = None
    for box in results.boxes:
        cls_name = model.names[int(box.cls[0])]
        if cls_name in ("cup", "bottle", "cell phone"):
            x1, _, x2, _ = box.xyxy[0]
            target_zone = zone_for_x(int((x1 + x2) / 2))
            break

    if target_zone == stable_zone and target_zone is not None:
        stable_count += 1
    else:
        stable_zone, stable_count = target_zone, 0

    if stable_count == 10:  # ~10 consecutive frames = confirmed, not flicker
        ser.write(f"{stable_zone}\n".encode())
        ser.read_until(b"D")  # block until the arm acks completion
        stable_count = -1000  # cool down until a new object appears

Calibration & risk notes

Where this actually fails

Not detection — pretrained YOLO on cup/bottle/phone just works. It's grasp reliability: cheap servos have backlash, and a gripper that closes with inconsistent force drops things. Run the day-3 grasp gate before building anything on top of it.

Power

Servos draw current spikes that brown out an MCU if sharing its 5V rail. Separate supply, common ground — this single fix resolves most "the arm resets randomly" bugs.

Lighting

Calibrate zone boundaries and test detection under the exact lighting you'll demo in. A different room's glare or shadow can shift things enough to need re-tuning on the day.

Project B

TinyML Sound Anomaly Detector

An ESP32 with an I2S microphone runs an on-device classifier — trained in Edge Impulse — that listens for specific sound events (glass breaking, a distress call) against a background-noise baseline, and fires a buzzer/LED/notification the moment it hears one. No moving parts, inference runs entirely on the microcontroller.

Architecture

I2S mic streams audio → ESP32 extracts MFCC features on-device → trained neural network (exported from Edge Impulse) classifies the window → if top class ≠ "background" and confidence exceeds threshold, fire alert with a debounce window so one event doesn't retrigger repeatedly.

8-day timeline

01

Order & wire

Order parts. Pick your 3 classes precisely — e.g. glass-break, distress-call, background-normal. The background/normal class is mandatory: without it the model will treat any sound as an event. Wire the INMP441 to the ESP32's I2S pins (well-documented, standard pinout).

02

Prove the audio path

Flash Edge Impulse's data-forwarder firmware, confirm raw audio streams cleanly into Edge Impulse Studio over serial. This connectivity check is the main "does the hardware work" gate — clear it early, not on day 6.

03

Collect data (part 1)

Record samples in the actual room you'll demo in, at the actual distance and volume you'll demo at. Label clips in Edge Impulse Studio as you go, rather than batching labeling for later.

04

Collect data (part 2)

Aim for 3–5 minutes of audio per class minimum. Deliberately record the room's own background chatter/hum as the "normal" class — this is what actually suppresses false alarms later.

05

Build & train the impulse

MFCC/MFE feature block + a small neural network classifier in Edge Impulse Studio. Train, read the confusion matrix. Whichever class confuses another, go back and collect more/cleaner samples for that pair — this iteration is normal, budget real time for it.

06

Deploy to hardware

Export as an Arduino library (EON compiler, for a smaller flash footprint). Flash to ESP32. Wire the buzzer/LED (and OLED if used) to fire on the top predicted class above a confidence threshold — start around 0.8.

07

Live-tune in the real room

Test repeatedly in the actual demo environment. Tune the confidence threshold against false positives. Add a 3–5 second debounce so one event doesn't spam the buzzer. Explicitly test that normal talking doesn't trigger "distress."

08

Buffer + polish

Final test pass. GitHub README with the Edge Impulse project link, confusion matrix screenshot, wiring diagram. Rehearse explaining MFCC features and why on-device inference (not cloud) is the point of a microcontroller project.

Code skeleton

Edge Impulse's exported library does the feature extraction and inference; your code is the thin layer around it that decides when to act.

// esp32_sound_alert.ino — skeleton around an Edge Impulse export
#include <YOUR_PROJECT_inferencing.h>  // generated by Edge Impulse export
#include <driver/i2s.h>

const float CONFIDENCE_THRESHOLD = 0.80;
const unsigned long DEBOUNCE_MS = 4000;
unsigned long lastAlertAt = 0;

void fireAlert(const char *label) {
  unsigned long now = millis();
  if (now - lastAlertAt < DEBOUNCE_MS) return; // swallow repeat triggers
  lastAlertAt = now;
  digitalWrite(BUZZER_PIN, HIGH);
  Serial.printf("ALERT: %s\n", label);
  delay(500);
  digitalWrite(BUZZER_PIN, LOW);
}

void loop() {
  signal_t signal;
  // ... fill `signal` from the I2S ring buffer per Edge Impulse's example ...
  ei_impulse_result_t result;
  run_classifier(&signal, &result, false);

  float best = 0; const char *bestLabel = "background";
  for (size_t i = 0; i < EI_CLASSIFIER_LABEL_COUNT; i++) {
    if (result.classification[i].value > best) {
      best = result.classification[i].value;
      bestLabel = result.classification[i].label;
    }
  }
  if (strcmp(bestLabel, "background") != 0 && best >= CONFIDENCE_THRESHOLD) {
    fireAlert(bestLabel);
  }
}

Data & risk notes

Where this actually fails

Not the hardware — there are no moving parts to break. It's data: too little background-noise coverage means real-room hum gets misclassified as an event. Budget two full days for recording, not one rushed evening.

Confidence threshold is your main dial

Too low and it false-alarms on every cough; too high and it misses real events. Tune this in the actual demo room, not your bedroom.

Demo-day checklist

  • Test in the actual demo room's ambient noise level beforehand, not just at home
  • Confidence threshold locked in the night before, not adjusted live
  • Power bank charged; USB cable as backup
  • A rehearsed, deliberately triggered "event" sound ready to demonstrate on command
